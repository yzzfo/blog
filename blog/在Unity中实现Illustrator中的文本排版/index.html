最近需要在Unity中实现和Illustrator中一样的文本渲染效果，但Unity的TextMeshPro和Illustrator差别还是蛮大的，所以在这记录一下相关的内容。

在横版的情况下

如何确定字体高度

  有时候我们将fontSize设置为12px的时候却发现他的preferHeight却是14px，那么这是怎么来的呢？为了明白这个问题，我们需要先知道字体中的三个基础属性:

  1. UPM：Units per Em，通常为1000，fontSize将对应该值，可以理解为标准字体大小
  2. Ascent: 上界距离基线的距离
  3. Descent: 下界距离基线的距离

Ascent与Descent组合在一起用于确定字体最大的高度Height=Ascent-Descent，preferHeight对应该值。所以可以得到height_px = Height / UPM * fontsize_px，这点在Unity和Illustrator中是一致的。

例如：UPM=1000，Ascent：852 Descent：-248 ，那我们可以知道该字体大小为 (852+248)/1000 = 1.1 em即当fontSize设置为10pt的时候，他实际大小为11pt。

如何确定字体宽度

  宽度和高度基本同理，但宽度的数据在每个字的内部，即Width=glyph.metrics.horizontalAdvance。 width_px = Width / UPM * fontsize_px

如何确定字体位置

  确定了字体大小也只是确定了包围盒,但位置并不知道，他以哪个地方为锚点开始画也不知道，这个时候就轮到了baseline这个属性，根据公式Top = (Ascent-baseline)/Height和Bottom = (baseline-Descent)/Height，可以算出baseline所在的位置，bottom_offset = height_px * Bottom，从底部向上偏移bottom_offset这么多就是baseline的位置，字体在设计的时候也是根据baseline进行设计的，这样就确定了位置。

行高

Unity

  TextMeshPro LineHeight=0的时候默认使用的是，TextMeshProAsset中的LineHeigt属性以下称为T_LineHeight，T_LineHeight属性单位和Ascent是一样的，所以计算px的算法也是一致的，lineHeight_px = T_LineHeight / UPM * fontSize_px。LineHeight单位为1em/100，即lineHeight_px = T_LineHeight / UPM * fontSize_px + LineHeight * fontSize / 100

Illustrator

  行高就是在位置属性中设置的大小，比如12pt那么一行就是12pt，即换行文字向下偏移12pt。

文字高度

Illustrator

文字高度不同于行高（行高指偏移，文字高度指单行时元素高度），此处文字高度指如下图所示的文字框的高度。

下边界

在Illustrator中，如下思源黑体之所以出现如此大的高度原因在于其字体内存在glyph，该glyph的ymin比Descent更小，因此导致框的下边界下移，即: 若字体中glyph的ymin小于Descent，则导致下边框下移，要解决这种问题可以尝试将哪些过高的glyph删除。

上边界

对于上边框，此处采用 https://www.glyphrstudio.com/app/ 该软件中的数值为基准进行计算，影响上边界的属性为Capital letter height，从baseline到上边界的距离符合h = 0.6*fontSize + 0.5*capital_letter_height/UMP，若文本本身高度超过了该计算值，则该值为文本高度。

总上所述，文字高度总公式为h = 0.6*fontSize + 0.5*capital_letter_height/UMP - Descent/UPM * fontSize,该公式适用于一般不超过默认边界线的文本，若出现超出的字符则按照相应规则对其中的数值进行替换处理。

踩坑点

  在探索上边界规则的时候一开始使用的FontCreator来查看修改的字体数据，但其导出的数据似乎存在兼容性等问题有些字体数据符合预期有些又是不符合预期的，导出的字体在glyphrstudio查看数据发现有些字体与修改的数据不一致，通过在glyphrstudio修改数据后在到Illustrator中进行校验才能算出符合预期的结果。How does Illustrator calculate text frame height 这篇文字中指出OS/2相关的数据，也可能是FontCreator中这部分数据存在问题导致输出字体存在差异。

在竖版的情况

  竖版情况下Unity和Illustrator兼容的部分相当有限，这里就记录以下每个部分的渲染逻辑。

Illustrator

罗马排布形式

  罗马排布形式即字母符号不会旋转90度。

文字高度/行高

文字大小依赖如下属性:

  1. UPM
  2. glyphRect

利用UPM来算出字体高度的最小值，glyphRect算出实际高度，例如j会比UMP大一些，a比UMP小，此时得到的高度a为UPM，j为glyphRect算出的高度，换而言之渲染时若字符小于1em则让其占用1em高度，若大于1em，则按实际高度。
例外，若有字体中有Vertical属性，就以(Vertical_Ascent - Vertical_Descent)为基准。

非罗马排布

文字高度/行高

同罗马排布形式，但区别在于字母和符号，因为会旋转所以长宽需要互换。

行距

行距离1000为1em，例如fontSize=12pt，1000即为12pt。

Unity

  Unity本质上是没竖版这类型的，可以单字绘制，大小和位置可以通过模仿Illustrator中规则来进行制作。

其他

Unity和Illustrator读取的字体属性可能不一致，有时候例如思源黑体，Unity中的字要比Illustrator中大的多，在FontCreator查看可以发现，思源黑体的Typo Ascender和Ascender数值不一样，AI和Unity应该是读取了不同的属性导致两者发生差距，在手动修改Unity的Asset值后，AI与Unity的字体可以一致显示，或者在FontCreator中修改两者的属性值。Illustrator用的Typo Ascender
