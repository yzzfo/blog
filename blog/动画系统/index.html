动画系统

Unity中常用的Avatar有两种，一种是Generic另一种是Humanoid，Generic类型通常用于自定义骨骼，或者说如果有专门的一套骨骼制作则用Generic会好一些，Generic就是当纯的顶点映射，骨骼节点同名即可进行动画播放。Humanoid则采用骨骼映射的形式进行播放，通过将骨骼A映射到Unity的一个标准人体模型，通过这个中间层来映射给别的Humanoid骨骼进行播放。 今天遇到一个问题就是现在有两个fps游戏模型素材，一个是第三人称用的射击动画，另一个是第一人称用的射击动画，他们两个都是用的同一套骨骼结构，但却无法在第一人称动画导入中使用import avatar from other导入第三人称的avatar来直接进行映射。

第三人称和第一人称的骨骼如图所述，均为一样的结构，且模型本身也是由用一个mesh所组成，所以预想的方式为直接将第一人称的avatar设置为第三人称所生成的avatar，设置之后出现了问题，骨骼映射出现了巨大的Y向偏差，导致模型直接超出Skinmesh的渲染范围，到地底去了，如图所示

产生该偏差的原因可能是第一人称动画本身就存在Hip节点的偏移，加之映射的缩放导致的

之后选择用同一个模型进行Create From this mode操作来创建avatar

  1. 打开第一人称模型的配置后会发现其及其混乱，这是由于第一人称的动作本身就扭曲导致的，仔细观察可以发现左边的arm都已经穿到右侧去了

  2. 照着骨骼顺序将右侧的Unity标准骨骼结构填入，此处可通过保存第三人称的Mapping，在Load到第一人称简化操作，但在这里进行该操作后只能Mapping出上半身的映射，下半身没映射上去，不知是何原因，剩下一半就手动给拖进去。

        

  1. 填充好节点之后就是将模型摆成T-Post，这里通常可以通过Pose -> Enforce Pose来摆成T-Pose，但Enforce Pose在这次的模型中无效，应该也是由于第一人称模型过于扭曲的缘故。

  2. 既然Enfore Pose无效那么怎么来摆T-Pose呢，由于第三人称模型和第一人称模型骨骼一模一样，而且第三人称模型并不扭曲在avatar中正规的T-Pose模型，于是我们可以去他的Avatar Config中将他的模型拷贝出来，做一个T-Pose预制体，用来将里面各个节点的坐标赋值给第一人称的avatar。

  3. 复制Avatar工具

    using System;
using System.Collections;
using System.Collections.Generic;
using UnityEditor;
using UnityEngine;

public class AvatarCopy : EditorWindow
{
    [MenuItem("Copy/Avatar")]
    public static void ShowWindow()
    {
        EditorWindow.CreateInstance<AvatarCopy>().Show();
    }

    public Transform from;
    public Transform to;
    public void OnGUI()
    {
        from = EditorGUILayout.ObjectField(new GUIContent("From"), @from, typeof(Transform)) as Transform;
        to = EditorGUILayout.ObjectField(new GUIContent("To"), to, typeof(Transform)) as Transform;

        if (GUILayout.Button("Copy"))
        {
            //广度优先
            foreach (var node in to.GetComponentsInChildren<Transform>())
            {
                var f = Find(node.name, @from);
                if (f)
                {
                    node.localScale = f.localScale;
                    node.localRotation = f.localRotation;
                    node.localPosition = f.localPosition;
                }
            }
            EditorUtility.SetDirty(to);
        }
    }

    public Transform Find(string name, Transform root)
    {
        foreach (Transform child in root.GetComponentsInChildren<Transform>())
        {
            if (child.name == name)
            {
                return child;
            }
        }
        return null;
    }
}

  4. 在第一人称的Avatar Config界面中将模型拖入To中，将第三人称的T-Pose，预制体拖入From中点击Copy即可

    

7.在场景中查看效果，可以发现动画Y轴的偏移对了，但肩膀位置不对，没有还原出第一人称时的扭曲

  1. 要还原扭曲就得知道通常骨骼动画是怎么做的，通常在骨骼动画中我们只会对各个关节进行旋转，通过旋转来达到位移，也就是说骨骼是不会拉伸的。但在第一人称的动画中，我们可以发现，这肩膀是拉伸了的，也就是说，关节不但可以旋转，还位移了，这中东西在Generic动画系统中并没什么问题，应该Generic仅仅就是映射点，并不管其他的东西，但在Humanoid中默认骨骼是不允许位移的，这是就得开启Translation DoF选项，也就是开启位移这个自由度。第一人称和第三人称均要开启。

Humanoid的原理测试

假设我们有两个Humanoid的Avatar A和B 均为正规的T-Pose，现在我们想让B来播放A的动作，即A映射到B身上去，正常情况下B将正常播放B的动作，此时我们将A的Avatar的根节点向后转一转，即后仰个20度并保存，此时将会导致B前倾20度，这是由于A和B均通过Unity的中间骨骼即Avatar进行映射，可能是如下过程，由于A当前设置后仰20度的状态为他的T-Pose，即标准状态，但动画中A还是直立的，所以从T-Pose到动画中的状态需要A进行前倾20度，于是乎A要完成这个动作就变成了前倾20度并完成该动画片段，将这条指令映射到B上就导致B前倾了20度。

同样我们不改动A，来改动B，将B的Avatar前倾20度，最终结果将是B前倾20度，这是因为前倾20度就是B的标准状态，一切动作都是依照标准状态来做的，那么A在直立时做的标准动作，到了B这就是B前倾20度来做这动作。

关于Copy avatar from other 和 Create from this的猜测，估计Avatar中记录了骨骼的长度映射关系，对于不同的第三人称模型，使用同一个avatar均没问题，这便可以排除骨骼中的位置旋转的影响，他们唯一没变的就是骨骼长度，而第一人称由于骨骼拉伸，这也导致骨骼与第三人称不匹配，所以推测avatar保存的是骨骼长度映射关系，由于这个关系导致了偏移，在上面的案例中若同时开启DoF，使用Copy的动画将会有个很巨大的放大效果，估计也是这个原因导致的。
