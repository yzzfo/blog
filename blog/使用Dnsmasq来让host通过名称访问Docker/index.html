使用Dnsmasq来让host通过名称访问Docker.

  最近在使用Docker的时候遇到了这种问题，首先我想让这个容器能够自由的开放端口供外部访问(例如V2ray的管理面板)，其二我还需要访问其他容器例如(Nginx Proxy)，要完成第一个任务通常使用host模式即可，完成第二件事通常就是创建一个功能的network接口，所有容器都使用这个network接口互相通过域名访问。但现在遇到的需求则是需要同时满足两者的特性，这时候我们可以通过使用Dnsmasq来实现这个。

思路

配置Dnsmasq容器jpillora/dnsmasq，让其开放53端口供外部访问，同时使用一个network接口A，其他的容器例如alist,webdav这类希望通过反代访问的容器，也使用network A接口，这样在dnsmasq中我们便能通过容器名称访问到这些容器了，但在默认情况下改容器的/etc/resolv.conf并未配置正常的DNS可以用于解析外部域名例如google.com，所以需要在里面加上一条nameserver 1.1.1.1这样改容器自身便能进行DNS解析了，之后配置Dnsmasq，no-resolv要删除，仅保留server=127.0.0.11这一个server，其他server都删除，这样便配置完成了。

第二步，设置我们的目标容器为host模式，这样我们便能够自由开放端口，为了通过名称访问其他的容器，我们还需要让本机的dns解析走Dnsmasq去解析，参考设置DNS进行DNS设置，这样我们在宿主机便能通过容器名称访问容器了。

整体DNS解析流程类似这样

host -> localhost:53 -> dnsmasq -> container /etc/resolv.conf -> docker dns
                                                               \ -> 1.1.1.1

Adguard

当使用Dnsmasq时若遇到查询不存在的容器名称时会导致出现DNS查询的死循环，其提供的dns-loop-detect使用后又无法查找本地的DNS，在改为使用Adguard后可以解决这个问题，原理和之前的Dnsmasq一样,DNS中添加127.0.0.11(11为Docker内置的DNS),但在设置时需要选择最快的IP地址这个DNS查询选项，只有这个才能保证Adguard在查询时会去调用127.0.0.11,其他两个选择不能确保必定走127.0.0.11进行查询。

Dnsmasq和Adguard组合

  最近在使用qbittorent的是否发现docker-proxy内存占用过高了，参考这个教程,进行了一下排查发现是adguard导致的，重启几次后发现他的内存占用还是缓慢上升，估计是产生了类似dnsmasq的循环查询导致的，将其改为并发查询后内存有所缓解，但依旧还是维持在比较高的水平，再将127.0.0.11:53给删除之后重启，内存保存在了正常水平，为了解决host模式下解析容器ip的问题最终选择采用adguard+dnsmasq的形式。

排查时使用的命令

ps -ax|grep docker-proxy

adguard+dnsmasq 结构拓扑

在宿主机容器中修改resolve.conf或者对特定程序例如nginx在解析时指定127.0.0.2时即可
