协同 Nginx 工作

由于服务器上同样有 Nginx，所以 80 和 443 自然是不能给 frp 的，但为了使 Frp 能穿透域名服务所以需要在 Nginx 下反代对应域名至 frp 中，在 nginx proxy manager 中可以通过直接声明类似如下的条目来指定匹配的域名 server_name "~.*-frp\.host\.com"; 该设置将能将该正则表达式匹配到的域名都转发至 frp 中进行穿透。

穿透内网 Https

PVE 系统的管理界面是强制 Https 访问的，但这样无法通过 type=http 或 type = https 直接穿透，需要使用插件 https2http 或者 tcp 穿透来解决，但 tcp 需要自己开一个新的端口，为了使用域名又得在 NPM 中单读开启一个反代才能访问，所以不选择 tcp。使用如下的配置方式便可以将内网的 https 服务穿透出去，且对外暴露的是 http 协议，这在 NPM 的视角下也统一了访问端口都是 80。

[pve-dashboard]  
type = http  
custom_domains = [pve.example.com](http://pve.example.com/)  
plugin = http2https  
plugin_local_addr = 10.10.10.3:8006  
plugin_host_header_rewrite = 10.10.10.3  
plugin_header_X-From-Where = frp  
use_compression = true  
use_encryption = true

后续发现 https2http 还是会导致 wss 无法连接，但 Caddy 的 Https 转 Http 不会导致这种情况

NPS

nps 也是一个不错的内网穿透工具，相对于 frp 来说他有一个很不错的 UI 管理界面，他还能通过服务端控制内网设备穿透哪一个端口，即在 frp 中需要客户觉得暴露什么端口，但在 nps 中可以由服务端来决定客户端开启什么端口。

在使用 nps 的时候发现 PVE 穿透会出现 502 的报错，问题应该和 frp 中是一样的，也需要类似 frp 中 https2http 这样的插件才能解决，但 nps 项目以及停更许久所以又改用 frp 了。

使用 Caddy 将 Https 转为 Http

http://192.168.3.246 {
    file_server
    encode zstd gzip
    reverse_proxy * https://192.168.3.2:8006 {
        header_up Host 192.168.3.2
        header_up X-Real-IP {http.request.remote.host}
        header_up X-Forwarded-Port {http.request.port}
        header_up X-Forwarded-Proto {http.request.proto}
        transport http {
            tls_insecure_skip_verify
        }
    }
    # tls /config/cer/server.pem /config/cer/server.key
}

https://forum.proxmox.com/threads/connection-error-401-no-ticket-nginx-reverse-proxy.94158/

这种方法在本地情况下只能通过 127.0.0.1 访问，即便设置了 host 以及 scheme 只 https 也还是没效果，可能还有别的东西需要设置吧，但在 VPS 上通过 NPM 反代该端口通过 https 进行访问则可以正常进入 PVE 界面。

User (http) -> Caddy (http) -> PVE (https) 无法登陆
User (https)-> VPS(http) -> NPS(http) -> Caddy (http) -> PVE (https) 正常使用

在 PVE 中，登陆时会发送 Cookie 至服务端，但 Cookie 的发送条件是当前为 https/localhost 才会发送，即该功能是前端限制，故无法通过 nginx 设置 header 解决，可以通过添加证书来处理这个问题。
